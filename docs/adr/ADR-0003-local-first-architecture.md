## 背景

drawzu の目的を、当初の「図を触ってスキーマが出る」から一段掘り下げ、
**「人間が図で見ているDBの姿と、AIがコードを書く時に前提にしているDBの姿を、
同じ1個のモデルに揃えること」** と定めた(2026-07-09の対話)。

この目的から、モデル(ADR-0001)に対する「窓」が2種類必要になる:

- **人間の窓**: ブラウザで開くER図ビュー。どこが何に繋がっているかを見る。
  テーブル名やカラム名を直すとモデルに反映される
- **AIの窓**: MCPツール(`get_model` / `patch_model`)。AIは設計をモデルに流し込み、
  コードを書く時はモデルを読んで前提にする

どちらの窓からの変更も、もう一方に即座に見える必要がある(AIが `patch_model` したら
開きっぱなしのブラウザの図が動く)。この「即時に揃う」体験が成立する実行形態を選ぶ。

## 決定

```
      対象プロジェクト(例: customer-base)で動く 1 つの Node プロセス
      ┌─────────────────────────────────────────────┐
      │            Schema Model(真実の源・1個)          │
      │      ↑↓ WebSocket           ↑↓ MCP(stdio)     │
      │  人間: ブラウザのER図ビュー   AI: Claude Code     │
      │      永続化: <対象プロジェクト>/.drawzu/model.json │
      └─────────────────────────────────────────────┘
```

- **実行形態**: MCPサーバーとWebビュー配信を同じプロセスに同居させる。
  両方の窓が同一プロセス内の同一モデルを参照するため、同期処理そのものが不要
- **モデルの置き場所**: 対象プロジェクトのリポジトリ内 `.drawzu/model.json`。
  スキーマ設計の履歴がプロジェクト本体のgit履歴と同じ場所に残る
- **実DBからの取り込み**: 専用の introspect 機能は作らない。AIは既に migration ファイルや
  Supabase 接続経由で実DBを読めるので、読んで理解した内容を `patch_model` で流し込む。
  「AI自身が取り込み役」とすることで、drawzu 側にDB接続・SQLパーサを持たずに済む

### 実装フェーズ

| フェーズ | 作るもの | 状態 |
|---|---|---|
| Phase 1 | モデル型定義 + ローカルWebビュー(ER図・ライブ同期・ファイル永続化) | 実装済み |
| Phase 2 | MCP化(`get_model` / `patch_model` / `to_sql`、stdio) | 実装済み(モデルを持たない薄いブリッジ `server/mcp.ts` が HTTP API に中継する形) |
| Phase 3 | リモート化(任意。候補: Cloud Run / Fly.io。WebSocket対応が条件で Vercel は不適) | 構想のみ |

Phase 3 を閉ざさないため、Phase 1 の時点でコアを通信路非依存に書き、
モデルの読み書きをストレージ層として1枚抽象化しておく。

### ADR-0002 との関係

ADR-0002 の「ER図エディタ本体(フルスタックGUI)は作らない」は維持する。
Phase 1 のWebビューは、既存プロトタイプ(`prototype/schema-canvas.jsx`)を土台にした
**薄いビューア+軽い編集(名前変更・カラム属性変更程度)** に限定し、
成熟済みエディタ群(drawDB等)と競う機能(undo/redo、自動レイアウト、大規模図の最適化等)には
踏み込まない。また ADR-0002 のMCPツール表のうち introspect は本ADRにより「作る機能」から外れる。

## 却下した選択肢

- **B(リモートホスト+クラウドストレージ)**: Claude Code 以外(claude.ai のカスタムコネクタ等)
  からも繋げる利点はあるが、認証・デプロイ・マルチテナントの実装が先に必要になり、
  モデルが対象プロジェクトのgit履歴から切り離される。目的(想定を揃える)に対して
  今は過剰。Phase 3 として余地だけ残す。なおホスト候補の比較では、サーバーレスで
  WebSocket を張れない Vercel は不適、コンテナ常駐の Cloud Run / Fly.io が適
- **C(既存GUIを窓に借りる)**: ADR-0002 時点の初期案。drawDB 等との接続には
  モデル⇔DBML/JSON の変換層と、相手側の状態管理への追従が必要で、
  「AIの patch が即座に図に出る」ライブ同期の成立が相手実装に依存する。
  見る+名前を直す程度なら自前の薄いビューの方が確実で小さい
